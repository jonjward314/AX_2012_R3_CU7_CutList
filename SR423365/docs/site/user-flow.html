Step-by-step flow

  1. User opens McaCutlistRecords.
  2. Default filter applies: unprocessed records (IsCutlistProcessed = No).
  3. User narrows scope (Model, Prefix, Build Schedule, Cut Group, SalesLine, etc.).
  4. LoadRecords maps form filters into create query and runs cutlist creation action.
  5. FilesReports passes current query scope into reports/files form.
  6. Reports/files pipeline runs and can mark selected rows as processed.

Interaction map

Form filter layer

User decides scope in grid filters and processed checkbox.

Create layer

Build/BOM context writes rows into McaCutlist with quantity fields, scheduling, and grouping IDs.

ReportDP layer

Transforms selected McaCutlist rows into McaCutlistTmp for each report design.

SSRS layer

Applies expressions (sum, ceiling, per-series normalization) and renders stakeholder-visible values.

Why logic matters to stakeholders

  • The same selected record scope can produce different values between PROD and DEV because formulas changed.
  • DEV introduces per-series handling and rounded totals in key report regions.
  • Rows marked processed may disappear from default form view unless processed filter is toggled.
