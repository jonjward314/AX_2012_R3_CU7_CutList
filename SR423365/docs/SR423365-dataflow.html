1) System Overview

The cutlist subsystem is centered on table McaCutlist. Records are created from BOM context, enriched with shop-floor routing metadata (machine, move-to area, cut group, blueprint, etc.), then reused by machine-file generation and SSRS reports.

This write-up is intentionally based on Production Code artifacts only (not Current Dev Code), so behavior described here matches what is deployed in this snapshot.

SalesLine + exploded BOM context (via McaBOMSearchCutlist)
↓
Create engine: McaCutlistCreate / McaCutlistCreate_Module
↓
Persistent table: McaCutlist
↓
Report DP: McaCutlistReportDP → TempDB McaCutlistTmp → SSRS PDF
↓
File/Report orchestrator: McaCutlistFilesReports + McaCutlistFile + McaCutlistReports
Important: In the current Production Code snapshot, Pieces is actively used across create/report/file paths. A field named Per_Series is not present in these Production Code table/class definitions.

3) Record Creation Flow (Most Important)

3.1 Orchestration path

McaCutlistCreateService.process(contract)
  -> processCutlist(contract)
      -> McaCutlistCreate.run() or createMultiTasks() (batch)
         -> processCutlistRecords()
            -> for each SalesLine in query:
               -> McaBOMSearchCutlist.run()
               -> iterate McaBOMExplodeTmp
               -> doCutlistItems()
                  -> createCutlistRecord() for each matched BOM raw-material line

3.2 BOM selection strategy in doCutlistItems()

Creation attempts progressively broader BOM lookup paths until rows are found:

  1. Primary: Item coverage + transfer-warehouse site path.
  2. Fallback 1: parent BOM site transfer-warehouse path.
  3. Fallback 2: parent BOM site direct path.
  4. Fallback 3: any BOM version path.

All paths apply BOM effective-date filtering against off-line/required date:

(BOM.fromDate <= offLineDate OR !BOM.fromDate)
AND
(BOM.toDate   >= offLineDate OR !BOM.toDate)

3.4 Module-based creation variant

McaCutlistCreate_Module extends the base create class, but sources rows from McaCutlistModuleTmp and assigns module-centric identifiers (for example using CutModuleId / converted SalesLineRefRecId behavior).

4) Output Generation: Machine Files + SSRS

4.1 High-level orchestrator

McaCutlistFilesReports.run() validates query safety, preserves the original query, then executes in sequence: 

runFiles()  -> McaCutlistFile.run()
runReports() -> McaCutlistReports.run()
runLabels()  -> McaCutlistLabels.run() [optional]
runMarkProcessed() -> McaCutlistReports::markProcessed(query)

4.2 Machine file behavior

McaCutlistFile uses cutlist parameters for output folders: MachineFilePath (sheet) and MachineFilePathLinear (linear). It updates generated filename metadata back to McaCutlist.MachineFileName.

4.3 Report behavior

McaCutlistReports can print Summary, Cover Sheet, and Detail versions of McaCutlistBaseReport. It prepares destination folder when printing to file and executes report controllers by report design.

4.4 Data provider transformation

McaCutlistReportDP.processReport() routes to Cover/Detail/Summary processors, each reading McaCutlist and writing display-ready rows into TempDB table McaCutlistTmp (names, formatted dimensions, grouping keys, etc.).

In detail path, Pieces is transformed to an expanded quantity: real2int(cutlist.Pieces * cutlist.KitQty).

6) Important Logic Notes

  • Batch scalability: create service can split work into runtime tasks (one per sales line) when executed in batch.
  • Validation guardrails: file/report orchestrator validates user query inputs before proceeding.
  • Processed state: records are marked processed after output actions via McaCutlistReports::markProcessed(query).
  • Usage calculation: table methods such as usageCalcRounding() influence report values.
  • Delete path: dedicated delete service iterates selected rows and deletes in a transaction with deadlock retry.

7) Troubleshooting Trace (Practical)

If a cutlist row is missing or output is wrong, follow this order:

  1. Verify create query selection (build schedule/sales lines) reaching McaCutlistCreateService.
  2. Confirm McaBOMSearchCutlist returns expected rows in McaBOMExplodeTmp.
  3. Check which doCutlistItems() fallback branch was used.
  4. Inspect raw inserted McaCutlist row: quantities, machine, move-to, cut group, prefix.
  5. Validate report or machine-file query filters (especially usage type and cut group).
  6. Confirm McaCutlistParameters file paths and output permissions.
  7. If rows disappear from active worklist, check IsCutlistProcessed updates after output run.
For SR423365-specific analysis: focus first on where Pieces is set in creation and expanded in detail/report/file paths.
